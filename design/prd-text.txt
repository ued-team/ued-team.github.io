Description 專案簡介與需求來源
1. 業務與商品特性驅動
日遊商品相較票券商品更複雜，涉及：
● 多方案與差異比較
● 集合資訊與交通安排
● 成團、天候、時間、使用限制等不確定因素
● 現場履約品質高度依賴供應商與導遊 / 司機執行
因此單靠轉換率、訂單結果、事後評論，無法完整反映使用者在決策前與履約中的真實
問題。
2. 內容治理與前台優化需求
目前團隊正在推進商品 / 方案內容治理與前台優化，希望確認：
● 哪些資訊對使用者最重要
● 哪些資訊雖已補充，但仍未真正被理解或感知
● 內容治理是否有實際改善決策品質與購買信心
需要透過更細緻的 feedback，建立「治理項目 ↔ 使用者感受 ↔ 成效指標」之間的對應
關係。
3. 營運與風險管理需求
在體驗當下，平台需要比事後評論更早掌握：
● 集合失敗
● 導遊 / 司機支援不足
● 行程延誤或調整未被妥善處理
● 商品描述與現場體驗落差
這類問題若能在旅程中提早識別並介入，有機會降低：
● 最終不滿
● 客訴升級
● 社群擴散
● 品牌與公關風險
4. 現有資料不足
目前可取得的資料多為：

● 行為數據：瀏覽、點擊、下單、取消
● 被動回饋：客服工單、申訴、退款
● 事後評價：評論、低分回饋
缺口在於：
● 決策前的疑慮不夠明確
● 體驗中的問題無法即時捕捉
● feedback 難以直接回扣到治理面向與後續 action
因此針對 KKday 日遊商品 user journey 中兩個高影響節點，建立更可行動的 user
feedback 機制，補足現有行為數據與事後評論之間的資訊斷層，協助產品、內容治理、
營運與供應商管理團隊更即時且更精準地掌握使用者需求與風險。
本專案將聚焦兩個節點：
1. 評估與比較
使用者在商品頁、方案頁與比較決策過程中，面對日遊商品資訊複雜、方案差異
多、限制條件多的特性，常有「不知道還缺什麼資訊才能下決定」或「頁面雖有資
訊但未被有效理解」的情況。
本專案希望透過結構化 feedback，更清楚辨識：
a. 使用者最在意哪些資訊
b. 目前資訊呈現是否足以支撐決策
c. 哪些內容治理與前台優化仍未真正解決問題
2. 體驗當下
日遊商品屬於長時段履約型體驗，從集合、找導遊、交通移動、景點停留、天候變
化、延誤調整到現場應變，都可能影響使用者感受。若平台僅在旅程結束後才透
過評論或客訴得知問題，往往已錯失介入時機，並可能演變為大客訴、社群負評
或公關事件。
本專案希望建立體驗中的 feedback 與風險感知能力，用於：
a. 及早發現履約異常
b. 支援客服 / 營運及時介入

Spec 概覽
關鍵 5 要素
1. 觸發時機（Trigger Design）👉 問對時機，比問什麼更重要
a. 評估與比較
i. 只在「有猶豫訊號」時問：
1. 停留久
2. 切換方案
3. 回訪商品
ii. 避免一進頁就問（干擾決策）
b. 體驗當下
i. 要「分段問」：
1. 集合後（最重要）

-- 16 of 27 --

template ver: 20220511
2. 行程中段
ii. 不要只在結束後問（太晚）
iii. 核心原則：問在「使用者還來得及被幫助」的時候
2. 問題設計（Question Design）👉 目標不是收意見，而是「可行動分類」
a. 評估與比較
i. 問「卡點」，不是滿意度
ii. 一定要可分類：
1. 行程內容
2. 價格
3. 集合
4. 成團
iii. 重點：每個選項都要能對應到治理面向
b. 體驗當下
i. 問「是否有問題」＋「問題類型」
ii. 題目要極短（1–2題）
